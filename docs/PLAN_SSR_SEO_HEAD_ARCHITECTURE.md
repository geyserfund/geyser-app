# SSR SEO Head Architecture Plan

## Objective

Ensure social crawlers (Discord, X, Slack) and search crawlers receive correct SEO metadata from server-rendered HTML for public routes, without changing functional behavior for users.

## Implementation Status (March 2026)

- Phase 1 foundation is implemented with `react-helmet` static extraction and server template head injection.
- Shared head tags now include baseline OG + Twitter (`twitter:title`, `twitter:description`, `twitter:image`) and canonical fallback.
- Project bootstrap now runs a render-time query path, so project metadata can be fetched during SSR.
- `react-helmet-async` migration remains a hardening follow-up, not a blocker for current SSR SEO output.

## Current Gap

SSR now serializes and injects head tags, but full consistency still depends on continued migration of remaining client-late public data paths.

## Architecture Constraints

This plan follows `docs/ARCHITECTURE.md`:

- Keep module/domain ownership for metadata logic.
- Keep page/view components thin and UI-focused.
- Keep shared head infrastructure in `src/config`/`src/shared`.
- No loader/meta route migration in this phase.

## Head Library Decision

### Option A: Stay on `react-helmet` (minimal patch)

Use `Helmet.renderStatic()` and inject output in server template.

- Pros: lowest scope.
- Cons: global side effects, not request-safe under concurrent SSR.

### Option B: Move to `react-helmet-async` (next hardening)

Use request-scoped `HelmetProvider` with per-request context extraction on SSR.

- Pros: request-safe, better long-term SSR correctness.
- Cons: slightly larger change set.

### Option C: Router-level metadata API (future)

Would require broader routing/data refactor and is out of scope.

### Final Decision

Proceed in two steps:

1. Phase 1 now: **Option A (`react-helmet`)** for minimal, low-risk rollout.
2. Follow-up: **Option B (`react-helmet-async`)** for request-scoped concurrency safety.

## Metadata Feature Parity (Current -> Required)

Current `Head` supports:

- `title`
- `og:title`
- `og:description`
- `meta[name="description"]`
- `og:image`
- `og:url`
- `og:type`
- `twitter:card`
- `twitter:site`
- canonical link
- child tags (JSON-LD scripts)

Phase 1 must preserve all of the above.

Planned improvements:

- Add `twitter:title`, `twitter:description`, `twitter:image` for better preview consistency.
- Add optional `og:image:alt`.
- Keep JSON-LD injection parity.

## Target State

For all public SSR routes:

- Final HTML `<head>` contains route metadata and JSON-LD where defined.
- Metadata is generated by module-owned builders, not ad-hoc tags in random components.
- Funding/refund/private CSR behavior remains unchanged.

## Implementation Phases

## Phase 1: SSR Head Pipeline (Foundation)

1. Keep `react-helmet` and extract SSR head via `Helmet.renderStatic()`.
2. Extend SSR result payload to include serialized head fragments.
3. Inject head fragments into `index.html` via dedicated marker (with fallback before `</head>`).
4. Preserve Apollo SSR (`getDataFromTree` + `__APOLLO_STATE__`) unchanged.
5. Keep fallback behavior when no route-specific tags exist.

Deliverable: crawlers receive server HTML with real metadata tags.

## Phase 2: Project Route Reliability

`ProjectLayout` already sets project-specific head data; Phase 2 hardens deterministic behavior and fallback quality.

1. Standardize project metadata builder in module tools.
2. Define strict fallback chains for title/description/image/canonical.
3. Validate route variants:
  - `/project/:name`
  - `/project/:name/prelaunch`
  - `/project/:name/draft`
  - `/project/:name/posts/:postId`
  - `/project/:name/rewards/:rewardUUID`
4. Verify JSON-LD remains present in final HTML where expected.

Deliverable: project links consistently produce rich previews.

## Phase 3: Reusable SEO Architecture Pattern

Create a reusable contract so all modules can add metadata safely.

1. Add shared SEO contract:
  - `src/shared/seo/types.ts` (`SeoDescriptor`)
  - `src/shared/seo/SeoHead.tsx` (descriptor -> head tags)
  - `src/shared/seo/url.ts` (request-safe origin/canonical helpers)
2. Add module-level builder pattern:
  - `modules/project/tools/seo/*`
  - `modules/discovery/tools/seo/*`
  - `modules/grants/tools/seo/*`
3. Keep pages thin:
  - pages call builder + render `SeoHead`
  - avoid hand-written OG blocks in many places

Deliverable: repeatable metadata architecture for all public pages.

## Phase 4: Coverage Expansion

Prioritize public routes that still rely on client-late data or default metadata:

1. Landing sections with delayed client rendering.
2. Discovery/heroes sections that fetch Airtable content in effects.
3. Any public page inheriting only app default head metadata.

Deliverable: broad and consistent SSR SEO coverage.

## Caching and Correctness

1. Keep current HTML/Apollo cache strategy initially.
2. Ensure head output participates in cached HTML payload.
3. Prefer request-derived origin for canonical/og:url.
4. Avoid metadata depending on client-only state where possible.

## Validation

1. Automated:
  - unit tests for metadata builders and fallback chains
  - SSR integration tests asserting head tags on representative routes
2. Manual:
  - page-source check for representative routes
  - Discord link preview verification
3. Regression:
  - private/funding/refund flows remain CSR
  - no hydration mismatch increase
